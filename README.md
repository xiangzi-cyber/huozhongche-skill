# huozhongche-skill

一个用于“AI火种车数智课堂”系列文章与推文写作的通用写作系统仓库。

它不只是一份 `Codex Skill`，也是一套可以跨平台复用的写作方法包。你既可以把它安装进 Codex，也可以把它当成适用于 ChatGPT、Claude、Kimi、豆包、元宝等模型的提示词与流程规范来使用。

## 仓库内容

- `SKILL.md`
  - Codex 版本的主说明文件，定义完整写作流程与规则。
- `scripts/dedup_check.py`
  - 用于表达去重分析的辅助脚本。
- `references/`
  - 预留参考资料目录。
- `templates/`
  - 提示词模板与后续可复用模板目录。
- `templates/universal-writing-prompt.md`
  - 适用于多平台模型的通用写作提示词模板。
- `docs/skill-understanding.html`
  - 系统结构理解说明页面。
- `docs/editor-training-deck.html`
  - 面向协作老师的系统讲解版页面。
- `docs/editor-training-deck.md`
  - 对应讲稿版说明。
- `docs/project-background.md`
  - 项目背景与命名更新说明。
- `docs/universal-usage.md`
  - 非 Codex 场景下的通用使用说明。
- `examples/`
  - 精选案例目录，用于放示例稿、对照稿和训练材料。
- `examples/material-intake/`
  - 前置素材收集表案例，包含表格样例与飞书多维表格搭建说明。

## 项目背景

`AI火种车` 目前已统一更新命名为：`AI火种车数智课堂`。

根据项目背景信息，这个公益项目：

- 由 WaytoAGI 教育版发起
- 旨在通过 AI 志愿老师将 AI 教育普及至偏远地区，缩小城乡数字鸿沟
- 项目首航于 2025 年 11 月在云南扎根，深入多民族聚居区开展“在地化”教学

因此，本仓库可以理解为：围绕 `AI火种车数智课堂` 项目所沉淀的一套内容写作与协作规范。

更多背景说明可见：

- `docs/project-background.md`

## 这套系统解决什么问题

它不是单纯教“怎么把活动写出来”，而是一套从素材梳理到终稿交付的写作系统，重点包括：

- 信息梳理与缺口识别
- 故事主线提炼
- 结构与标题风格选择
- 初稿深化与平视视角
- 配图策略与隐私保护
- 表达去重、错别字检查与终稿交付

更具体地说，它主要帮助解决以下几类问题：

- 素材很多，但不知道从哪里开始写
- 现场信息不完整，容易边写边猜
- 同一个系列写多了，文章越来越像
- 写完正文后，配图和结构还要重新返工
- 协作写稿时，大家对“什么是合格的 AI火种车数智课堂 风格”理解不一致

## 核心特点

- 把写稿过程拆成清晰的 6 个步骤，减少协作时的混乱感。
- 强调“平视视角”，避免怜悯叙事和扶贫标签化表达。
- 把配图建议纳入写作流程，而不是发稿前临时补图。
- 通过表达去重机制，避免系列稿件语言风格越来越像。

## 适用场景

这套系统特别适合以下场景：

- 写“AI火种车数智课堂”项目相关的公众号文章、系列推文、阶段性报道
- 根据活动素材表、课表、采访记录、现场照片整理成正式稿件
- 多位老师或协作者共同参与写作，需要统一文章风格和检查标准
- 已经有一批发布过的稿件，希望新稿继续保持系列感，但又不重复
- 需要在写作过程中同步考虑配图选择、隐私保护和图文节奏

## 适合谁使用

- 项目内容编辑
- 协作写稿老师
- 系列内容统稿人
- 需要快速接手“AI火种车数智课堂”写作体系的新伙伴

## 不适合的场景

以下情况不太适合直接使用这套系统，或者需要先做额外调整：

- 完全不属于“AI火种车数智课堂”项目语境的通用营销文案
- 只需要一句短宣传语、海报标题或极短社媒 copy
- 没有素材表、没有活动信息、没有可核实背景的纯空写任务
- 不需要系列一致性、也不需要图文协同的普通随笔型写作

## 典型使用时机

你可以把它理解成一套“从接收素材到交付成稿”的写作流程。比较典型的使用时机包括：

- 活动刚结束，手头已有素材表和部分照片，准备开始写首稿
- 首稿写完，准备补标题方向、配图建议和文风统一
- 新稿准备上线前，需要和既往已发布稿件做表达去重
- 团队内部需要向新协作者解释“AI火种车数智课堂写作系统到底在要求什么”

## 使用方式

这套系统现在支持三种使用方式：

### 1. Codex Skill 模式

适合已经在用 Codex 的同学。

使用方式：

1. 把仓库内容安装到本地 Codex skill 目录。
2. 在写作任务里调用这套 skill。
3. 一并提供素材表、课表、人物信息、历史稿件和图片说明。

这一模式的优势是：

- 可以直接按 `SKILL.md` 的流程工作
- 可以调用 `scripts/dedup_check.py` 做辅助查重
- 更适合长期、稳定、批量化使用

### 2. 通用 AI 模型模式

适合使用 ChatGPT、Claude、Kimi、豆包、元宝、通义等模型的同学。

使用方式：

1. 打开 `templates/universal-writing-prompt.md`
2. 将提示词复制到你常用的模型对话框中
3. 一并上传素材表、照片说明、历史稿件和补充背景
4. 先让模型完成“信息梳理与缺口识别”，再继续进入写稿阶段

这一模式的关键不是“一次性让模型直接出终稿”，而是让它按这套系统的顺序推进。

### 3. 人工协作检查模式

适合团队在飞书文档、共享表格或内部培训里使用。

使用方式：

1. 先统一使用前置素材收集表
2. 按六步流程逐项检查
3. 把“缺口信息、故事主线、标题方向、配图建议、去重检查”分别作为协作节点

这一模式下，即使不依赖任何特定模型，也能把写作标准统一起来。

## 不同平台都能复用的核心部分

以下内容不依赖 Codex，因此都可以直接跨平台复用：

- 六步写作流程
- 平视视角与表达原则
- 配图策略与隐私保护规则
- 前置素材收集表逻辑
- 已发布案例的结构参考
- 通用提示词模板

真正偏向 Codex 的部分主要只有两项：

- `SKILL.md` 的自动调用方式
- `scripts/dedup_check.py` 这样的本地脚本能力

也就是说，就算不用 Codex，这个仓库里大部分方法依然成立。

## 推荐工作流

一个常见的使用方式是：

1. 先通过前置素材收集表，把活动基础信息、课表、故事积木和补充字段整理齐
2. 用这套系统做信息梳理与缺口识别
3. 确定主线、辅助线、叙事结构和标题方向
4. 完成初稿，并用平视视角、隐喻、细节等原则深化
5. 把配图建议直接嵌入稿件
6. 结合已发布稿件做表达去重，再进行错别字和一致性检查
7. 最后交付终稿与修改说明

## 实际使用时建议一起提供的材料

无论你使用的是 Codex，还是其他模型，推荐都尽量补齐以下输入：

- 素材表，如 `.xlsx`
- 活动日期与课表
- 关键人物姓名与身份
- 已发布的相关文章或历史稿件
- 可用照片或图片说明
- 项目背景或学校背景补充说明

这些信息越完整，这套系统的效果越稳定。

如果是团队协作场景，推荐先统一使用前置素材收集表，再进入正式写稿流程。

## 案例上传建议

如果后续要把写稿案例也上传进仓库，建议优先采用“精选案例”而不是一次性上传大体量原始包。

更推荐的方式是：

- 每次上传 1-3 篇代表性案例
- 优先上传可读版本，如 `.md`、`.pdf` 或体量适中的 `.docx`
- 如果需要保留原始大包，建议放云盘链接，而不是直接塞进 GitHub 仓库
- 案例文件可以按“场景 / 学校 / 阶段 / 类型”分类，方便后续检索和培训使用
