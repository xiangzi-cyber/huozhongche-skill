# huozhongche-skill

一个用于"AI火种车数智课堂"系列文章与推文写作的通用写作系统仓库。

## 更新日志

### 2025-04(素材整理模块上线)

本次更新补全了写作系统中「写稿前」的完整素材整理链路,解决了之前最大的痛点:素材散落在群聊里、看图找信息全靠人工。

**新增内容:**

- `docs/material-collection-guide.md` - 素材收集说明书,适合任何没有技术背景的协作老师,指导如何收集、整理、传递素材

- `docs/workflow-simplified.md` - 简化版工作流(适合新手主编),AI做整理和写稿,你做搬运和决策
- `docs/workflow-technical.md` - 技术版工作流(适合有CLI基础的用户),通过飞书CLI实现更高程度的自动化
- `templates/material-agent-prompt.md` - 素材整理助手系统提示词,适配任何支持看图的AI平台(扣子/GPT/Claude等)
- `examples/material-intake/universal-template-v3.xlsx` - 通用素材收集表模板(飞书/Excel通用)
- `examples/material-intake/coze-agent/hadapu-example.xlsx` - 扣子Agent素材整理输出示例(哈达铺案例)

**对整体流程的意义:**

这套写作系统原本的起点是"已有整理好的素材表"--但真实情况是老师一般只会往群里发照片和几句话,没人有时间填表格。这次更新把这个缺口补上了:从「一张群里的照片」到「可以喂给写稿系统的结构化素材」,现在有了完整的工具和说明书。

这套内容的最佳使用方式,是把它整体安装到支持 Skill 或自定义智能体知识包的系统中使用。当前仓库提供了可直接落地的 `Codex Skill` 版本,同时也保留了跨平台迁移能力,方便在 ChatGPT、Claude、Kimi、豆包、元宝等模型中复用。

## 3 步安装

如果你想先快速装起来,优先看这一段:

1. 获取仓库:克隆或下载本仓库到本地。
2. 放入 Skill 目录:把整个仓库目录放到你的 Skill 系统目录中,并保留 `SKILL.md`、`scripts/`、`templates/`、`docs/`、`examples/`。
3. 重启并验证:重启 Skill 系统后,用一句测试任务验证是否会先做"素材梳理 -> 缺口识别 -> 主线提炼"。

如果你使用的是 Codex,典型目录结构可以是:

```text
$CODEX_HOME/skills/huozhongche-skill/
```

如果你需要完整安装步骤、迁移建议和验证方式,请看:

- `docs/skill-installation.md`

## 项目背景

`AI火种车` 目前已统一更新命名为:`AI火种车数智课堂`。

根据项目背景信息,这个公益项目:

- 由 WaytoAGI 教育版发起
- 旨在通过 AI 志愿老师将 AI 教育普及至偏远地区,缩小城乡数字鸿沟
- 项目首航于 2025 年 11 月在云南扎根,深入多民族聚居区开展"在地化"教学

因此,本仓库可以理解为:围绕 `AI火种车数智课堂` 项目所沉淀的一套内容写作与协作规范。

更多背景说明可见:

- `docs/project-background.md`

## 这套系统解决什么问题

它不是单纯教"怎么把活动写出来",而是一套从素材梳理到终稿交付的写作系统,重点包括:

- 信息梳理与缺口识别
- 故事主线提炼
- 结构与标题风格选择
- 初稿深化与平视视角
- 配图策略与隐私保护
- 表达去重、错别字检查与终稿交付

更具体地说,它主要帮助解决以下几类问题:

- 素材很多,但不知道从哪里开始写
- 现场信息不完整,容易边写边猜
- 同一个系列写多了,文章越来越像
- 写完正文后,配图和结构还要重新返工
- 协作写稿时,大家对"什么是合格的 AI火种车数智课堂 风格"理解不一致

## 核心特点

- 把写稿过程拆成清晰的 6 个步骤,减少协作时的混乱感。
- 强调"平视视角",避免怜悯叙事和扶贫标签化表达。
- 把配图建议纳入写作流程,而不是发稿前临时补图。
- 通过表达去重机制,避免系列稿件语言风格越来越像。

## 适用场景

这套系统特别适合以下场景:

- 写"AI火种车数智课堂"项目相关的公众号文章、系列推文、阶段性报道
- 根据活动素材表、课表、采访记录、现场照片整理成正式稿件
- 多位老师或协作者共同参与写作,需要统一文章风格和检查标准
- 已经有一批发布过的稿件,希望新稿继续保持系列感,但又不重复
- 需要在写作过程中同步考虑配图选择、隐私保护和图文节奏

## 适合谁使用

- 项目内容编辑
- 协作写稿老师
- 系列内容统稿人
- 需要快速接手"AI火种车数智课堂"写作体系的新伙伴

## 不适合的场景

以下情况不太适合直接使用这套系统,或者需要先做额外调整:

- 完全不属于"AI火种车数智课堂"项目语境的通用营销文案
- 只需要一句短宣传语、海报标题或极短社媒 copy
- 没有素材表、没有活动信息、没有可核实背景的纯空写任务
- 不需要系列一致性、也不需要图文协同的普通随笔型写作

## 典型使用时机

你可以把它理解成一套"从接收素材到交付成稿"的写作流程。比较典型的使用时机包括:

- 活动刚结束,手头已有素材表和部分照片,准备开始写首稿
- 首稿写完,准备补标题方向、配图建议和文风统一
- 新稿准备上线前,需要和既往已发布稿件做表达去重
- 团队内部需要向新协作者解释"AI火种车数智课堂写作系统到底在要求什么"

## 使用方式与推荐优先级

这套系统现在支持三种使用方式,但推荐顺序并不相同。

推荐优先级如下:

- 第一优先:整体安装到 Skill 系统中使用,效果最好
- 第二优先:作为通用提示词与流程规范,迁移到其他模型中使用
- 第三优先:作为人工协作检查框架使用

### 1. Skill 系统整体安装模式(推荐)

适合已经在用 Codex,或正在使用其他支持 Skill、知识包、自定义智能体配置的平台的同学。

使用方式:

1. 把仓库内容整体安装到你的 Skill 系统或本地智能体目录中。
2. 在写作任务里调用这套系统。
3. 一并提供素材表、课表、人物信息、历史稿件和图片说明。

这一模式的优势是:

- 可以直接按 `SKILL.md` 的流程工作
- 可以调用 `scripts/dedup_check.py` 做辅助查重
- 前后文更稳定,不容易跳步
- 更容易把脚本、案例、模板和流程作为一个整体协同使用
- 更适合长期、稳定、批量化使用

当前仓库原生适配的是 Codex,但整体方法也可以迁移到其他支持类似 Skill 机制的平台中。

## 如何安装到 Skill 系统

如果你希望拿到最完整、最稳定的效果,推荐把整个仓库作为一个 Skill 包整体安装,而不是只复制其中一段提示词。

### 方式一:安装到 Codex

1. 下载或克隆这个仓库到本地。
2. 把整个仓库目录放到你的 Skill 目录中,并命名为 `huozhongche-skill`。
3. 确保目录里至少保留这些核心内容:
   - `SKILL.md`
   - `scripts/`
   - `templates/`
   - `docs/`
   - `examples/`
4. 重启 Codex,或重新加载本地 Skill。
5. 在新任务中直接提到这套系统,开始写作。

典型目录结构可以是:

```text
$CODEX_HOME/skills/huozhongche-skill/
├── SKILL.md
├── scripts/
├── templates/
├── docs/
└── examples/
```

### 方式二:安装到其他支持 Skill 或知识包的系统

如果你使用的是其他支持 Skill、知识包、自定义智能体目录的平台,建议也尽量保留整个仓库结构导入。

最低建议保留:

- `SKILL.md`
- `templates/universal-writing-prompt.md`
- `examples/material-intake/`
- `examples/phase-1-pdfs/`
- `docs/skill-understanding.html`

如果平台支持脚本或工具调用,也建议把 `scripts/dedup_check.py` 一并保留。

### 安装成功后的验证方式

安装完成后,可以直接用下面这类任务测试:

- "请按 AI火种车数智课堂写作系统,先梳理这份素材表并识别缺口。"
- "请不要直接写终稿,先给我 2-3 个主线方向和结构建议。"

如果系统能先做素材梳理、缺口识别、主线提炼,而不是一上来直接写成稿,就说明安装方向基本是对的。

### 2. 通用 AI 模型模式(可用,但次优)

适合使用 ChatGPT、Claude、Kimi、豆包、元宝、通义等模型的同学。

> 温馨提醒:
> 这套资料现在在 ChatGPT、Claude、Kimi、豆包、元宝等模型里都可以直接使用,先拿来跑流程、理解方法是完全没问题的。
> 不过,这一版更适合做"迁移使用"和"快速上手"。
> 如果后面想把效果跑得更稳定、更完整,最好还是放到带 Skill 系统的 AI 产品里。
> 那一层通常会更适合已经有一点 AI 编程或 AI 工作流基础,尤其对提示词组织、工作流拆分、知识包 / Skill 配置比较熟悉的人来使用。

使用方式:

1. 打开 `templates/universal-writing-prompt.md`
2. 将提示词复制到你常用的模型对话框中
3. 一并上传素材表、照片说明、历史稿件和补充背景
4. 先让模型完成"信息梳理与缺口识别",再继续进入写稿阶段

这一模式的关键不是"一次性让模型直接出终稿",而是让它按这套系统的顺序推进。

这一模式可以用,但整体稳定性通常不如完整安装到 Skill 系统中。

### 3. 人工协作检查模式(培训或兜底)

适合团队在飞书文档、共享表格或内部培训里使用。

使用方式:

1. 先统一使用前置素材收集表
2. 按六步流程逐项检查
3. 把"缺口信息、故事主线、标题方向、配图建议、去重检查"分别作为协作节点

这一模式下,即使不依赖任何特定模型,也能把写作标准统一起来,但它更适合作为培训、对齐或兜底方案,而不是首选执行方式。

## 不同平台都能复用的核心部分

以下内容不依赖 Codex,因此都可以直接跨平台复用:

- 六步写作流程
- 平视视角与表达原则
- 配图策略与隐私保护规则
- 前置素材收集表逻辑
- 已发布案例的结构参考
- 通用提示词模板

真正偏向 Skill 系统整体安装的部分主要有这些:

- `SKILL.md` 这样的系统级工作流入口
- `scripts/dedup_check.py` 这样的本地脚本能力
- 资料、模板、案例和流程在同一系统中的持续协同

也就是说,就算不在 Codex 里,这个仓库里大部分方法依然成立;但如果能整体安装到 Skill 系统中,使用体验和稳定性通常会更好。

## 推荐工作流

一个常见的使用方式是:

1. 先通过前置素材收集表,把活动基础信息、课表、故事积木和补充字段整理齐
2. 用这套系统做信息梳理与缺口识别
3. 确定主线、辅助线、叙事结构和标题方向
4. 完成初稿,并用平视视角、隐喻、细节等原则深化
5. 把配图建议直接嵌入稿件
6. 结合已发布稿件做表达去重,再进行错别字和一致性检查
7. 最后交付终稿与修改说明

## 实际使用时建议一起提供的材料

无论你使用的是 Codex,还是其他模型,推荐都尽量补齐以下输入:

- 素材表,如 `.xlsx`
- 活动日期与课表
- 关键人物姓名与身份
- 已发布的相关文章或历史稿件
- 可用照片或图片说明
- 项目背景或学校背景补充说明

这些信息越完整,这套系统的效果越稳定。

如果是团队协作场景,推荐先统一使用前置素材收集表,再进入正式写稿流程。

## 案例上传建议

如果后续要把写稿案例也上传进仓库,建议优先采用"精选案例"而不是一次性上传大体量原始包。

更推荐的方式是:

- 每次上传 1-3 篇代表性案例
- 优先上传可读版本,如 `.md`、`.pdf` 或体量适中的 `.docx`
- 如果需要保留原始大包,建议放云盘链接,而不是直接塞进 GitHub 仓库
- 案例文件可以按"场景 / 学校 / 阶段 / 类型"分类,方便后续检索和培训使用
