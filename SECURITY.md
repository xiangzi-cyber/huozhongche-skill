# Security Policy

## Supported Version

当前仓库以默认分支 `main` 上的最新内容作为受支持版本。

如果你发现的是旧分支或历史提交中的问题，也欢迎报告，但修复会优先在 `main` 上进行。

## Reporting a Vulnerability

请不要通过公开 issue 报告安全问题。

推荐的报告方式：

1. 如果仓库已经开启 GitHub 的私密漏洞报告功能，请使用仓库 `Security` 页面中的私密报告入口。
2. 如果该入口暂时未开启，请通过仓库维护者的 GitHub 联系方式私下联系，并说明这是安全问题。

为了帮助快速定位，请尽量附上：

- 受影响的文件、目录或功能
- 复现步骤或最小复现示例
- 可能造成的影响
- 是否涉及凭据泄漏、账号风险或未公开资料
- 如有建议修复方向，也欢迎一并说明

## Expected Response

维护目标如下：

- 3 个工作日内确认收到报告
- 7 个工作日内给出初步处理状态
- 在修复和披露节奏上尽量与报告者协同

## Contributor Safety Rules

为了降低仓库被攻击或被误提交敏感信息的风险，提交者请遵守以下规则：

- 不要提交 token、cookie、`.env`、私钥、证书或任何账号凭据
- 不要提交未公开的个人信息、联系方式或原始敏感素材
- 不要在未审查权限的情况下新增 GitHub Actions、自动化脚本或外部集成
- 新增案例文件前，先确认这些材料已经可以公开，或已经做过去标识化处理
- 涉及安全变更时，优先走 PR 审核，不要直接推送到 `main`

## Operational Notes

这个仓库是持续运营项目的一部分，因此除代码安全外，也需要关注：

- 素材与案例文件中的隐私信息
- 协作账号权限是否过宽
- 默认分支是否允许无审查直接改动
- GitHub 后台安全能力是否已开启
