# 仓库安全加固说明

这份文档用于说明，当前仓库已经补了哪些基础安全防护，以及还有哪些关键项需要在 GitHub 后台手动开启。

## 这次已经补上的基础防护

仓库内已经新增了这些安全基线文件：

- `SECURITY.md`
  - 规范安全问题的私密报告方式，避免漏洞被公开 issue 暴露。
- `.gitignore`
  - 避免常见凭据文件、临时文件、私钥和本地处理产物被误提交。
- `.github/CODEOWNERS`
  - 为后续启用“Code owner review”提供基础。
- `.github/pull_request_template.md`
  - 把最关键的安全检查变成 PR 检查项。
- `.github/ISSUE_TEMPLATE/config.yml`
  - 在 issue 入口处提示安全问题不要公开提交。

这些改动能降低“误提交敏感内容”和“协作时漏检安全问题”的风险，但还不能替代 GitHub 后台的权限与扫描能力。

## 建议你立刻去 GitHub 后台开启的项目

以下项目最值得优先开启。

### 1. 保护 `main` 分支

建议在分支保护规则里至少开启：

- Pull request 后才能合并
- 至少 1 个审批
- Require review from code owners
- Dismiss stale approvals
- 禁止 force push
- 禁止删除受保护分支

这样可以显著降低默认分支被误改、被绕过审查或被恶意直接推送的风险。

### 2. 开启 Private Vulnerability Reporting

建议在仓库 `Security` 中开启私密漏洞报告。

这样研究者或协作者发现问题时，不需要公开开 issue，就可以私下报告给维护者。

### 3. 开启 Secret Scanning

如果仓库是公开仓库，建议开启 GitHub 的 secret scanning 相关能力。

它的作用是：

- 发现误提交的常见密钥
- 尽早提醒仓库维护者处理

如果你的账号或套餐支持 push protection，也建议一并开启。

### 4. 开启 Code Scanning

虽然当前仓库代码不多，但已经有 `scripts/dedup_check.py`，后续项目持续迭代时，代码量和风险面都可能增加。

建议在 GitHub `Security` 中为仓库开启 code scanning，优先使用默认配置。

### 5. 收紧 GitHub Actions 权限

当前仓库还没有 workflow，但如果后续新增自动化，建议先在仓库设置里把默认 workflow 权限改为只读。

同时建议：

- 新增 Actions 前先做权限审查
- 尽量少用不明来源的第三方 Action
- 如果必须使用第三方 Action，优先固定到明确版本或 commit

### 6. 定期复查协作者权限

持续运营项目最怕“权限一直开着没人回看”。

建议每月检查一次：

- 谁还有写权限
- 是否有人不再参与但仍保留权限
- 是否存在权限过大的机器账号或自动化账号

## 运营类项目额外要注意的点

这个仓库不是纯代码仓库，还包含案例稿、培训材料和素材结构说明，因此要额外注意：

- 上传案例前，确认内容已经公开或允许公开
- 不要把原始未清洗的采访记录、通讯录、账号信息放进仓库
- 如果 PDF 或文档来自外部处理流程，最好先检查元数据和附带信息
- 涉及学生、老师、学校的材料，优先上传公开版本或去标识化版本

## 建议的协作规则

为了让仓库在持续迭代中更稳，建议团队内部统一下面几条：

- 尽量通过 PR 合并，而不是直接推送 `main`
- 安全相关变更必须走 review
- 每次上传案例文件前，先做一次“是否公开可发”的确认
- 每次新增脚本或自动化前，先确认权限范围

## 每月 10 分钟安全检查清单

- 看一次仓库 `Security` 页面有没有新告警
- 看一次协作者列表有没有多余权限
- 确认 `main` 分支保护规则还在
- 确认私密漏洞报告仍然开启
- 回看最近合并内容里有没有误提交的临时文件或敏感信息

## 一句话总结

对于这个仓库，真正有效的安全加固不是只做一件事，而是同时做好三层：

1. 仓库内文件基线
2. GitHub 后台权限与扫描设置
3. 日常协作时的审查习惯
