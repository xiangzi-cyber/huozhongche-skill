# 通用使用说明

这份说明用于帮助非 Codex 用户也能使用这套“AI火种车数智课堂”写作系统。

不过需要先说明一点：

> 这套系统的最佳效果，仍然来自“整体安装到 Skill 系统中使用”。

这份文档更适合在暂时没有 Skill 安装条件时，作为迁移和备用方案来使用。

核心原则只有一句话：

> 不要把它理解成一个只能在某个平台里触发的技能，而要把它理解成一套优先适合整体安装、同时也可以跨工具迁移的写作方法。

## 适用对象

这份说明特别适合：

- 使用 ChatGPT、Claude、Kimi、豆包、元宝、通义等模型的协作者
- 在飞书文档里多人协作写稿的老师
- 需要先统一方法，再选择具体工具的项目伙伴

## 推荐优先级

如果条件允许，推荐按下面的顺序使用：

1. 整体安装到 Skill 系统中
2. 迁移到通用大模型中
3. 作为人工协作检查框架使用

原因很简单：

- Skill 系统更容易保持流程完整
- 不容易漏掉前置素材梳理和缺口识别
- 更适合把脚本、案例、模板和方法放在同一个工作流里
- 长期批量使用时更稳定

## 可以跨平台复用的部分

以下内容都不依赖 Codex，可以直接搬到其他平台：

- 六步写作流程
- 平视视角与表达原则
- 配图建议与隐私保护规则
- 前置素材收集表逻辑
- 历史案例作为表达与结构参考
- 通用提示词模板

## 更偏向 Codex 的部分

以下两项更偏向 Codex 本地环境：

- `SKILL.md` 的自动触发与技能安装方式
- `scripts/dedup_check.py` 的本地脚本执行

这不影响整体方法迁移。即使不在 Codex 里，依然可以照着同样的流程完成写稿；只是整体安装到 Skill 系统中的效果通常更好。

## 使用方式一：直接给任意模型

> 温馨提醒：
> 这套资料现在在 ChatGPT、Claude、Kimi、豆包、元宝等模型里都可以直接使用，先拿来跑流程、理解方法是完全没问题的。
> 不过，这一版更适合做“迁移使用”和“快速上手”。
> 如果后面想把效果跑得更稳定、更完整，最好还是放到带 Skill 系统的 AI 产品里。
> 那一层通常会更适合已经有一点 AI 编程或 AI 工作流基础，尤其对提示词组织、工作流拆分、知识包 / Skill 配置比较熟悉的人来使用。

当你暂时无法使用 Skill 系统时，最简单的做法是：

1. 打开 `templates/universal-writing-prompt.md`
2. 复制其中的提示词
3. 粘贴到你常用的模型对话框
4. 一并上传素材表、历史稿件、图片说明和补充背景

推荐的操作顺序不是“一次性要求出终稿”，而是分两轮：

第一轮：

- 让模型先做信息梳理
- 识别缺口
- 提出主线和结构方向

第二轮：

- 再让模型进入正式写稿
- 补标题方向
- 补配图建议
- 做文风统一和表达去重

## 使用方式二：飞书文档协作

如果你们团队更依赖飞书文档或共享文档，可以把这套系统拆成几个协作节点：

1. 前置素材收集
2. 缺口补充
3. 主线确定
4. 初稿生成
5. 配图建议补充
6. 去重和终稿检查

这样做的好处是：

- 大家不会一上来就焦虑“怎么写”
- 每一步都知道自己在解决什么问题
- 方便多人接力，而不是全都压在一个人身上

## 使用方式三：培训或内部对齐

如果你的目标不是立刻出稿，而是先把方法讲清楚，可以这样用：

- 用 `docs/editor-training-deck.html` 做讲解
- 用 `docs/skill-understanding.html` 帮大家理解结构
- 用 `examples/material-intake/` 解释为什么前置素材收集是系统起点
- 用 `examples/phase-1-pdfs/` 做成稿风格示例

## 推荐输入材料

无论在哪个平台里使用，建议尽量补齐这些材料：

- 素材表
- 活动日期
- 课表或流程安排
- 人物姓名与身份
- 原话、细节、故事积木
- 照片或图片说明
- 已发布历史稿件

输入越完整，输出越稳定。

## 推荐产出顺序

建议不要直接要“一篇最终可发稿”，而是按下面顺序拿结果：

1. 信息梳理
2. 缺口清单
3. 主线与结构建议
4. 初稿
5. 配图建议
6. 去重与终稿检查

## 一句话总结

这套仓库里的 `Codex Skill` 只是其中一种实现方式。

更底层的东西，其实是：

> 一套用于“AI火种车数智课堂”内容生产的写作流程、判断标准和协作方法。
